## Origin of the toolkit's approach

The "move directories in and out of `~/.claude/projects/` to scope `/insights`" approach comes from the **"Current workaround" section** of the feature request asking for native project-scoping: [anthropics/claude-code#23762 — Current workaround](https://github.com/anthropics/claude-code/issues/23762#current-workaround). The OP describes it as:

> Manually renaming/moving folders under `~/.claude/projects/` before running `/insights` to hide unwanted projects from the ingestion stage. This is fragile and tedious.

This toolkit is the automated version of that fragile/tedious workaround. The issue was closed as a duplicate of [#23311](https://github.com/anthropics/claude-code/issues/23311), and as of writing there's still no built-in scoping flag.

## Caching tradeoff: facets get re-extracted every run

The toolkit deletes `~/.claude/usage-data/session-meta/` and `~/.claude/usage-data/facets/` in three places: `backup-projects.sh`, between every project in `insights-per-project.sh`, and at the end of `restore-all.sh`. There's no "restore" step for these dirs because they're never backed up — only deleted, to force `/insights` to recompute against whatever set of projects is currently live.

### What's in those dirs

- **`session-meta/`** — local statistical summary per session (tokens, tools used, duration, git activity). Extracted from the JSONL logs locally, no API call.
- **`facets/`** — Claude-Haiku-generated qualitative tags per session (satisfaction level, goal type, agent performance). Each missing facet = **one Haiku API call** (capped at 8192 tokens per prompt).

### The cost

Deleting `facets/` makes `/insights` re-run AI extraction on every visible session (capped at 50 new per run). Because the toolkit clears facets *between every project* in the loop, no cross-project reuse happens. For N projects with ~M sessions each, you pay ~N×M Haiku calls per full toolkit run instead of paying once and reusing the cache.

Haiku is cheap so this is likely pennies, but it's not free, and it's not what you'd get from running `/insights` normally.

### Could the loop avoid this?

Possibly. Facets are keyed by session ID, so in theory the per-project loop could keep them across iterations — `/insights` would still re-process per-project because the *set of sessions* it sees changes, but it wouldn't re-call Haiku for sessions it's already classified. Worth doing only if the loop runs often enough that the bill matters.
